# 데이터 체커 프롬프트 아키텍처 설계 계획

## 1. 문서 목적

이 문서는 **데이터 체커: 기사 속 시각자료 근거 검증 서비스**의 프롬프트 구조를 정리한 설계 문서다.

문서의 목적은 다음과 같다.

- 에이전트별 역할을 명확히 한다.
- 각 에이전트가 읽는 입력과 작성하는 출력을 구분한다.
- `input_`, `ce_`, `ig_`, `vc_`, `merge_` prefix 규칙을 정리한다.
- 최종 판정 기준과 안전 표현 기준을 정의한다.
- 현재 구현된 `vc_` 최종판정 검토 에이전트와 향후 확장될 전체 구조를 연결한다.

## 2. 프로젝트 목적

이 프로젝트는 뉴스 기사 안에 포함된 표, 차트, 그래프, 도표 같은 시각자료가 기사 제목과 본문 주장을 제대로 뒷받침하는지 확인하는 서비스다.

이 서비스는 뉴스 전체를 보고 바로 가짜 뉴스라고 단정하지 않는다. 대신 다음을 확인한다.

- 표·차트가 기사 주장을 실제로 뒷받침하는가?
- 기사 제목이나 본문이 수치를 과장하고 있는가?
- 시각자료의 출처, 기간, 단위, 표본 수가 충분한가?
- 차트 구조가 독자에게 오해를 줄 가능성이 있는가?
- 현재 정보만으로 검증이 가능한가?

> **서비스 한 줄 요약:** 뉴스 기사 안의 표·차트와 기사 제목·본문 주장을 비교해 과장, 왜곡, 정보 부족 가능성을 찾아주는 AI 검증 서비스

## 3. 전체 에이전트 구조

최종 기획상 구조는 3개의 에이전트와 1개의 병합 노드로 구성한다.

```text
사용자 입력
→ input_ 상태 생성
→ ce_ 주장-근거 검증 에이전트
→ ig_ 검증정보 확인 에이전트
→ vc_ 최종판정 검토 에이전트
→ merge_ 결과 병합 노드
→ 최종 리포트
```

병렬 구조의 목표는 다음과 같다.

```text
input_
├─ claim_evidence_agent → ce_
├─ info_gap_agent       → ig_
└─ verdict_critic_agent → vc_

ce_ + ig_ + vc_
→ report_merge_node
→ merge_
```

단, 최종 구조를 실제로 연결할 때는 `verdict_critic_agent`가 `ce_`와 `ig_` 결과를 읽어야 하므로 실행 의존성을 명확히 설계해야 한다. 주장·근거 분석과 검증정보 확인은 병렬화할 수 있지만, 최종판정 검토는 두 결과가 준비된 뒤 실행하는 단계형 병렬 구조가 적절하다.

현재 구현 단계는 다음과 같다.

- `vc_` 최종판정 검토 에이전트가 먼저 구현되었다.
- `ce_`와 `ig_`는 아직 실제 에이전트가 아니라 임시 입력 기반으로 테스트 중이다.
- 현재 데모는 **vc_ 최종판정 검토 에이전트 + 임시 ce_/ig_ 입력 + Streamlit 서비스형 데모** 단계다.

## 4. prefix 규칙

| prefix | 의미 | 작성 주체 |
|---|---|---|
| `input_` | 공통 입력값 | 프론트/파서 |
| `ce_` | 주장-근거 검증 결과 | `claim_evidence_agent` |
| `ig_` | 검증정보 확인 결과 | `info_gap_agent` |
| `vc_` | 최종판정 검토 결과 | `verdict_critic_agent` |
| `merge_` | 사용자용 최종 리포트 | `report_merge_node` |
| `runtime_` | 시스템 상태, 오류, 경고 | 시스템 |

절대 규칙은 다음과 같다.

1. `input_` 변수는 모든 에이전트가 읽을 수 있지만 수정하지 않는다.
2. `ce_` 에이전트는 `ce_` 변수만 작성한다.
3. `ig_` 에이전트는 `ig_` 변수만 작성한다.
4. `vc_` 에이전트는 `vc_` 변수만 작성한다.
5. 병합 노드만 `merge_` 변수를 작성한다.
6. 서로 다른 prefix의 값을 임의로 수정하지 않는다.
7. 뉴스가 가짜라고 직접 단정하지 않는다.
8. 없는 정보는 지어내지 않는다.

## 5. 공통 입력 변수

모든 에이전트는 `input_` 변수를 읽을 수 있다.

| 변수명 | 의미 |
|---|---|
| `input_news_title` | 뉴스 기사 제목 |
| `input_news_body` | 기사 본문 주장 |
| `input_chart_image_path` | 대표 시각자료 이미지 경로 또는 여러 경로를 줄바꿈으로 합친 문자열 |
| `input_chart_image_paths` | 여러 시각자료 이미지 경로 리스트 |
| `input_chart_text` | 뉴스 내부 시각자료에서 읽은 보조 수치·설명 |
| `input_source_text` | 시각자료 출처, 주석, 단위, 축 설명, 범례 등 |

`input_chart_text`는 기사 본문에 적힌 수치를 복사하는 칸이 아니다. 뉴스 내부에 실제로 삽입된 표, 차트, 그래프, 도표에서 읽은 수치나 설명을 입력하는 보조 값이다.

## 6. 최종 판정 기준

`vc_recommended_judgement`는 현재 아래 4개 값만 사용한다.

| 판정 | 의미 |
|---|---|
| 대체로 뒷받침됨 | 표·차트가 기사 제목과 본문 주장을 대체로 뒷받침하는 경우 |
| 주의 필요 | 일부 근거는 확인되지만 기사 표현이 과장되었거나 해석 범위가 넓은 경우 |
| 검증 제한 | 출처, 기간, 단위, 비교 기준 등 핵심 정보가 부족해 판단이 어려운 경우 |
| 왜곡 가능성 높음 | 표·차트 내용과 기사 주장이 명확히 어긋나는 경우 |

기존 기획 문서의 “믿어도 됨”을 “대체로 뒷받침됨”으로 바꿀지는 팀 논의 후 결정한다. 현재 구현에서는 **대체로 뒷받침됨**을 유지한다.

### 6.1 과도한 판정 방지 원칙

- 정보가 부족하면 강한 긍정·부정 판정보다 `검증 제한`을 우선 고려한다.
- 일부 근거가 있지만 기사 표현이 강하면 `주의 필요`를 우선 고려한다.
- `왜곡 가능성 높음`은 시각자료에서 확인되는 내용과 기사 주장이 명확히 어긋날 때만 사용한다.
- 보조적으로 확인하면 좋은 정보만 부족한 경우에는 이를 이유로 `대체로 뒷받침됨`을 무조건 `검증 제한`으로 낮추지 않는다.

## 7. “대체로 뒷받침됨” 판정을 받을 수 있는 기사 조건

> **핵심 정의:** “대체로 뒷받침됨”은 뉴스 전체가 무조건 믿을 만하다는 뜻이 아니다. 기사 제목과 본문 주장이 뉴스 내부 시각자료가 직접 보여주는 범위 안에서 대체로 확인된다는 의미다.

### 조건 1. 기사 주장과 시각자료가 거의 일치해야 한다

시각자료가 2021년부터 2024년까지 전기차 충전소 수 증가를 보여준다면, 기사 제목은 “전기차 충전소 3년간 꾸준히 증가”처럼 시각자료가 보여주는 범위 안에서 작성되어야 한다.

반면 “전기차 충전 인프라 부족 완전 해소”는 충전소 수 증가만으로 직접 확인하기 어렵기 때문에 주의가 필요하다.

### 조건 2. 강한 표현이나 과장 표현이 없어야 한다

주의가 필요한 표현은 다음과 같다.

- 폭증
- 급락
- 압도적
- 완전 회복
- 사실상 사라졌다
- 역대급
- 대부분
- 확실히
- 때문에
- 원인

`대체로 뒷받침됨`에 가까운 표현은 다음과 같다.

- 꾸준히 증가
- 완만한 증가
- 소폭 증가
- 감소 흐름
- 증가세
- 하락세
- 전년 대비 증가

### 조건 3. 본문이 시각자료보다 넓게 해석하지 않아야 한다

시각자료가 “충전소 수 증가”와 “급속충전기 수 증가”만 보여준다면, 기사 본문은 “충전 인프라 수량이 확대되고 있다” 정도로 표현하는 것이 적절하다.

“충전 인프라 부족 문제가 완화되고 있다”는 전기차 대수 대비 충전기 수, 지역별 분포, 이용률, 대기시간 등의 추가 정보가 필요하므로 `주의 필요`가 될 수 있다.

### 조건 4. 시각자료의 기본 정보가 충분해야 한다

최소한 다음 정보가 확인되는 것이 좋다.

- 차트 제목
- 기간
- 단위
- 수치
- 비교 기준
- 출처 또는 자료 설명

핵심 정보가 여러 개 부족하거나 의미 있는 차트 사실을 확인하기 어려우면 `대체로 뒷받침됨`보다 `검증 제한`이 적절할 수 있다. 지역별 분포나 추가 세부 통계 같은 보조 정보만 부족하다면 판정을 무조건 낮추지 않고 추가 확인사항으로 제시한다.

### 조건 5. 여러 시각자료가 같은 방향을 보여야 한다

기사 주장이 “전기차 충전 인프라 수량이 늘고 있다”이고 시각자료 1이 충전소 수 증가, 시각자료 2가 급속충전기 수 증가를 보여준다면 `대체로 뒷받침됨` 가능성이 높아진다.

반대로 한 시각자료는 증가를, 다른 시각자료는 정체나 감소를 보여준다면 `주의 필요`가 될 수 있다.

> **핵심 정리:** 차트가 직접 보여주는 수치·기간·단위·추세 안에서만 기사도 주장하고, 폭증·완전 해소·원인·대부분 같은 강한 해석을 덧붙이지 않는 경우 `대체로 뒷받침됨`을 사용할 수 있다.

> **즉, 차트가 말하는 만큼만 기사도 말해야 한다.**

## 8. claim_evidence_agent 프롬프트 구조

예정 파일 위치:

- `third_agent/prompts/claim_evidence_prompt.md`
- `third_agent/agents/claim_evidence_agent.py`

역할은 뉴스 제목·본문 주장과 시각자료의 수치, 추세, 단위, 변화량을 비교하는 것이다.

읽는 값:

- `input_news_title`
- `input_news_body`
- `input_chart_image_path`
- `input_chart_image_paths`
- `input_chart_text`
- `input_source_text`

작성하는 값:

| 변수명 | 의미 |
|---|---|
| `ce_chart_facts` | 차트에서 확인한 사실 |
| `ce_claim_summary` | 기사 제목·본문 주장 요약 |
| `ce_strong_expressions` | 폭증, 급락, 완전 회복 등 강한 표현 |
| `ce_risk_flags` | 과장, 인과 단정, 기간 일반화 등 위험 신호 |
| `ce_draft_judgement` | 1차 판정 |
| `ce_draft_summary` | 1차 판정 이유 |

출력 원칙:

- `ce_` 키만 반환한다.
- 표·차트에서 확인할 수 없는 내용은 만들지 않는다.
- 기사 주장이 차트 범위를 넘어서는지 확인한다.
- 최종 단정 대신 1차 판정과 근거를 제공한다.

## 9. info_gap_agent 프롬프트 구조

예정 파일 위치:

- `third_agent/prompts/info_gap_prompt.md`
- `third_agent/agents/info_gap_agent.py`

역할은 시각자료를 해석하는 데 필요한 검증 정보가 충분한지 확인하는 것이다.

읽는 값:

- `input_news_title`
- `input_news_body`
- `input_chart_image_path`
- `input_chart_image_paths`
- `input_chart_text`
- `input_source_text`

작성하는 값:

| 변수명 | 의미 |
|---|---|
| `ig_metadata_status` | 정보 충분, 정보 일부 부족, 검증 제한 |
| `ig_found_info` | 확인된 출처, 기간, 단위 등 |
| `ig_missing_info` | 부족한 정보 목록 |
| `ig_limitation_reason` | 검증 제한 사유 |
| `ig_questions` | 사용자에게 물어볼 추가 확인 질문 |

출력 원칙:

- `ig_` 키만 반환한다.
- 맞다·틀리다를 직접 판정하지 않는다.
- 부족한 정보와 검증 제한 사유를 작성한다.
- 없는 정보는 지어내지 않는다.
- 핵심 판단을 막는 부족 정보와 보조적으로 확인할 정보를 구분한다.

## 10. verdict_critic_agent 프롬프트 구조

파일 위치:

- `third_agent/prompts/verdict_critic_prompt.md`
- `third_agent/agents/verdict_critic_agent.py`

역할은 최종 판정이 제공된 근거보다 강하게 단정되지 않았는지 검토하는 것이다.

읽는 값:

- `input_` 공통 입력
- `ce_` 주장-근거 검증 결과
- `ig_` 검증정보 확인 결과

작성하는 값:

| 변수명 | 의미 |
|---|---|
| `vc_recommended_judgement` | 추천 최종 판정 |
| `vc_unsafe_expressions` | 위험 표현 목록 |
| `vc_revision_needed` | 표현 수정 필요 여부 |
| `vc_revision_reason` | 수정이 필요한 이유 |
| `vc_safe_expression` | 사용자에게 제안할 안전한 표현 |
| `vc_critic_notes` | 검토 메모 |

중요 규칙:

- `vc_` 키만 반환한다.
- `input_`, `ce_`, `ig_` 값은 읽기만 한다.
- `merge_`, `runtime_` 값은 작성하지 않는다.
- 뉴스가 가짜라고 직접 단정하지 않는다.
- 위험한 표현이 있으면 더 안전한 표현으로 완화한다.
- 핵심 정보가 부족하면 강한 판정보다 `검증 제한`을 우선 고려한다.
- 보조 정보만 부족하면 긍정 판정을 자동으로 낮추지 않는다.
- 일부 근거가 있지만 표현이 강하면 `주의 필요`를 우선 고려한다.
- `왜곡 가능성 높음`은 시각자료와 기사 주장이 명확히 어긋날 때만 사용한다.

위험 표현 예시:

- 가짜 뉴스
- 조작
- 거짓
- 사기
- 완전히 틀림
- 명백한 허위
- 절대 믿으면 안 됨

안전한 표현 예시:

- 주의가 필요한 주장입니다.
- 검증에 제한이 있습니다.
- 과장 가능성이 있습니다.
- 왜곡 가능성이 높습니다.
- 현재 정보만으로는 충분히 뒷받침하기 어렵습니다.

구조화 출력 이후에도 코드 가드레일을 적용해 허용되지 않은 판정, 위험 표현, 핵심 정보 부족 및 출력 키 범위를 다시 확인한다.

## 11. report_merge_node 설계

파일 위치: `third_agent/nodes/report_merge_node.py`

역할은 `ce_`, `ig_`, `vc_` 결과를 사용자용 리포트로 변환하는 것이다. `vc_` 에이전트가 `merge_` 값을 작성해서는 안 되며, 사용자 화면에 보여줄 카드, 요약, 문제 지점, 부족 정보 그룹화는 병합 노드에서 담당한다.

작성하는 값:

| 변수명 | 의미 |
|---|---|
| `merge_user_facing_judgement` | 사용자에게 보여줄 최종 판정 |
| `merge_judgement_tone` | success, warning, info, error 등 UI 톤 |
| `merge_headline` | 한 줄 핵심 결론 |
| `merge_summary` | 사용자용 요약 |
| `merge_top_summary_cards` | 권장 판정, 문구 조정, 검증 신뢰도 카드 |
| `merge_issue_cards` | 문제 지점 카드 |
| `merge_evidence_cards` | 시각자료 근거 카드 |
| `merge_missing_info_cards` | 부족 정보 그룹 카드 |
| `merge_recommendations` | 권장 확인사항 |
| `merge_final_report` | 완성형 Markdown 리포트 |

현재 결과 화면 구성:

- 권장 판정
- 문구 조정
- 검증 신뢰도
- 결과 요약
- 문제 지점
- 시각자료 근거
- 상세 데이터

일반 사용자 화면에는 로컬 이미지 경로를 직접 노출하지 않는다. 시각자료는 실제 이미지와 사용자 친화적인 근거 설명으로 표시하고, 원본 경로는 상세 데이터의 raw state에서만 확인할 수 있게 한다.

## 13. vc_ 담당 범위의 완료 기준

`vc_` 최종판정 검토 에이전트는 아래 조건을 만족하면 담당 범위에서 완성에 가깝다고 본다.

- `vc_` 키만 반환한다.
- 위험 표현을 탐지한다.
- 위험 표현을 안전한 표현으로 완화한다.
- 최종 판정을 4개 중 하나로 제한한다.
- 핵심 정보 부족 시 과한 판정을 방지한다.
- 보조 정보 부족만으로 정상에 가까운 판정을 과도하게 낮추지 않는다.
- 일부 근거가 있으면 `왜곡 가능성 높음`을 남발하지 않는다.
- `vc_safe_expression`을 사용자에게 보여줄 수 있는 문장으로 생성한다.
- 병합 노드가 사용하기 좋은 안정된 JSON을 반환한다.
- 다양한 테스트 샘플에서 일관된 기준을 유지한다.

## 14. 테스트 샘플 기준

| 유형 | 기대 판정 |
|---|---|
| 정상에 가까운 기사 | 대체로 뒷받침됨 |
| 일부 해석 확장이 있는 기사 | 주의 필요 |
| 과장 표현 기사 | 주의 필요 |
| 정보 부족 기사 | 검증 제한 |
| 시각자료와 주장 충돌 기사 | 왜곡 가능성 높음 |
| 인과관계 단정 기사 | 주의 필요 |

테스트 시 다음을 확인한다.

- 판정이 너무 강하게 나오지 않는가?
- 핵심 정보 부족 상황에서 `검증 제한`이 적절히 나오는가?
- 보조 정보 부족만 있는 정상 사례가 과도하게 `검증 제한`으로 내려가지 않는가?
- 일부 근거가 있는 경우 `주의 필요`로 완화되는가?
- 위험 표현이 최종 출력에 남지 않는가?
- 사용자용 결과 화면에서 문제 지점과 부족 정보가 이해하기 쉽게 보이는가?
- 일반 사용자 화면에 로컬 이미지 경로가 노출되지 않는가?

## 16. 최종 정리

현재 프롬프트 아키텍처는 다음 방향으로 설계한다.

- `input_`는 공통 입력이다.
- `ce_`는 주장과 시각자료의 관계를 분석한다.
- `ig_`는 검증에 필요한 정보 부족 여부를 확인한다.
- `vc_`는 최종 판정이 과하게 단정되지 않도록 검토한다.
- `merge_`는 사용자에게 보여줄 서비스형 리포트를 만든다.

현재 구현은 전체 서비스 완성본은 아니지만, `vc_` 최종판정 검토 에이전트의 결과 출력 형식은 기획 방향에 맞게 서비스형 데모 수준까지 정리되었다. 이후 확장에서도 각 prefix의 작성 책임, 확인 가능한 근거만 사용하는 원칙, 보수적인 최종 판정 기준을 유지해야 한다.
